# Feature Specification: OpenClaw Enterprise — Secure Enterprise AI Assistant Platform

**Feature Branch**: `001-enterprise-ai-platform`
**Created**: 2026-03-13
**Status**: Draft
**Input**: User description: "OpenClaw Enterprise extends the existing OpenClaw open-source personal AI assistant with enterprise-grade capabilities: a hierarchical policy engine, information classification, graduated human-in-the-loop controls, agent-to-agent communication protocols, and intelligent task management — turning a personal AI assistant into an enterprise-ready platform."

## Clarifications

### Session 2026-03-13

- Q: What is the target user scale for initial release? → A: Up to 500 concurrent users per deployment.
- Q: Which connectors are required for MVP? → A: Gmail, GCal, Jira, GitHub, GDrive. Remaining connectors (Outlook, Linear, Notion, Confluence, GitLab) deferred to post-MVP.
- Q: How is initial data classification assigned? → A: Per-connector defaults + AI reclassification for sensitive content detection + admin override.
- Q: What is the task retention policy and lifecycle? → A: 90-day active retention. Completed tasks archived after 30 days, purged after 90. Lifecycle: discovered → active → completed → archived → purged.
- Q: What is the scope of cross-org agent-to-agent exchanges? → A: Intra-enterprise cross-org exchanges allowed, governed by org-level policies defining what data can be shared and with whom. Cross-enterprise exchanges blocked.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Policy-Governed Task Briefing (Priority: P1)

A knowledge worker starts their day and receives a prioritized daily briefing generated by their AI assistant. The assistant scans email, calendar, Jira, GitHub, and GDrive — plus Slack messages received through OpenClaw's existing Slack ChannelPlugin — discovers tasks, deduplicates across systems, scores by priority, and delivers a ranked list with time-block suggestions. Every action the assistant takes is governed by enterprise policy and logged to an immutable audit trail. (Note: Slack is accessed via OpenClaw's built-in messaging channel, not an enterprise connector plugin.)

**Why this priority**: This is the core value proposition — making knowledge workers more effective by surfacing what matters most. It also exercises the foundational layers (connectors, policy engine, audit) that all other features depend on.

**Independent Test**: Deploy a single-user instance with Gmail, Slack, Jira, GitHub, and Calendar connectors configured. Verify the assistant produces a morning briefing with correlated, deduplicated, priority-ranked tasks. Verify every connector read and model call is logged in the audit trail and every action passes through the policy engine.

**Acceptance Scenarios**:

1. **Given** a user has 12 unread emails, 28 unread Slack messages (via OpenClaw's existing Slack channel), 4 assigned Jira tickets, and 3 PR review requests, **When** the scheduled morning briefing triggers, **Then** the assistant produces a single prioritized task list with duplicates removed (e.g., a Slack message and a Jira ticket about the same work appear as one task).
2. **Given** an enterprise policy restricts the assistant to read-only access on GitHub, **When** the assistant discovers a PR review request, **Then** it includes the task in the briefing but does not take any write action on GitHub.
3. **Given** a user's calendar shows 5 meetings and a 2-hour free block, **When** the briefing is generated, **Then** it identifies the free block and suggests using it for the highest-priority deep-work task.
4. **Given** the policy engine is unreachable, **When** the assistant attempts to scan a connector, **Then** it denies the action and notifies the user that the briefing could not be generated due to a policy evaluation failure.

### User Story 2 — Graduated Auto-Response Engine (Priority: P2)

A knowledge worker enables auto-response for low-priority Slack messages and informational emails. The assistant classifies incoming messages, checks enterprise and team policies, and either responds autonomously (with notification), queues a draft for approval, or blocks the action — depending on the graduated autonomy level defined in policy.

**Why this priority**: Auto-response directly reduces communication overhead — the primary time sink for knowledge workers. It deeply exercises the policy engine's graduated autonomy model (autonomous / notify / approve / block), which is the core enterprise differentiator.

**Independent Test**: Send a variety of Slack messages and emails to a user with auto-response enabled. Verify the assistant classifies each correctly, checks policy, and takes the appropriate action (auto-respond, notify, queue for approval, or block). Verify all auto-responses are logged and reviewable.

**Acceptance Scenarios**:

1. **Given** a team policy allows auto-response for simple questions in internal Slack channels, **When** a colleague asks "What's the status of PROJ-123?" in an internal channel, **Then** the assistant checks Jira, composes a factual response, sends it with an AI disclosure label, and notifies the user.
2. **Given** an org policy blocks auto-sending of external emails, **When** the assistant classifies an incoming external email as needing a response, **Then** it drafts the response and queues it for the user's approval rather than sending it.
3. **Given** a user is in focus mode, **When** a direct Slack message arrives, **Then** the assistant acknowledges the message with a focus-mode notice (if policy allows).
4. **Given** an auto-response is sent, **When** the user reviews their activity log, **Then** every auto-response is visible with the original message, the generated response, the policy that authorized it, and the classification of the message.

### User Story 3 — Work Tracking Auto-Updates (Priority: P3)

An engineer merges a pull request on GitHub. The assistant detects the merge event via webhook, correlates it with a Jira ticket (using branch name or PR description), and automatically adds a summary comment to the Jira ticket and transitions it to "In Review" — all governed by policy and logged.

**Why this priority**: Engineers universally dislike manual Jira updates. Automating this provides immediate visible value and validates the webhook-driven event correlation architecture. It depends on connectors and policy from P1 but adds write actions governed by the approval model from P2.

**Independent Test**: Merge a PR whose branch name contains a Jira ticket key. Verify the assistant adds a comment summarizing the PR changes to the linked Jira ticket, transitions the ticket status, and links the PR. Verify the policy engine authorized each write action.

**Acceptance Scenarios**:

1. **Given** a PR with branch name `feature/PROJ-789-add-retry-logic` is merged, **When** the GitHub webhook fires, **Then** the assistant adds a comment to PROJ-789 summarizing the PR changes and transitions the ticket to "In Review."
2. **Given** a team policy allows Jira transitions to "In Progress" and "In Review" but blocks transitions to "Done," **When** the assistant detects a PR merge, **Then** it transitions to "In Review" but does not transition to "Done."
3. **Given** a PR references multiple Jira tickets in its description, **When** the PR is merged, **Then** each referenced ticket is updated independently.
4. **Given** the Jira connector has read-only permissions in the current policy, **When** the assistant attempts to update a ticket, **Then** the action is denied and the user is notified of the policy restriction.

### User Story 4 — Agent-to-Agent Information Exchange (Priority: P4)

Two knowledge workers each have an OpenClaw Enterprise assistant. One worker's assistant sends a status query to the other's assistant via Slack. The receiving assistant detects the OCIP metadata, checks policy and data classification, filters any information above the allowed classification level, responds within the configured round limit, and logs the full exchange on both sides.

**Why this priority**: Agent-to-agent communication is the most novel and complex capability. It requires the OCIP protocol, classification enforcement, loop prevention, and dual-sided audit logging. Building it after the core platform (P1-P3) ensures the foundational layers are stable.

**Independent Test**: Set up two OpenClaw Enterprise instances for two different users. Have Agent A query Agent B for a project status. Verify the OCIP metadata is present, classification filtering works (confidential data is excluded), round limits are enforced, and both sides log the exchange.

**Acceptance Scenarios**:

1. **Given** Agent A asks Agent B "What's the status of the data pipeline migration?", **When** Agent B has both internal and confidential context about the project, **Then** Agent B responds with only internal-classified information and withholds confidential details.
2. **Given** a policy sets a maximum of 3 exchange rounds, **When** the agents reach round 3 without resolution, **Then** the exchange is escalated to the human users with a summary of the conversation.
3. **Given** Agent B receives a request that requires committing its user to a meeting, **When** Agent B processes the request, **Then** it escalates to the human user for approval rather than responding autonomously.
4. **Given** two agents exchange messages, **When** either user reviews the audit log, **Then** the full transcript is available, including what data was shared, what was withheld, and which policies applied.

### User Story 5 — Org News Intelligence and Document Change Monitoring (Priority: P5)

A knowledge worker receives a personalized weekly digest of org-wide announcements, scored by relevance to their role, team, and active projects. Separately, when a watched document (architecture doc, security policy, runbook) changes substantively, the assistant detects the change, summarizes what changed and how it impacts the user, and delivers a notification based on urgency.

**Why this priority**: This capability adds significant value but is less foundational than task management, auto-response, and work tracking. It depends on the connector and policy infrastructure built in earlier phases.

**Independent Test**: Post several org-wide announcements to monitored Slack channels and email lists. Verify the assistant scores each for relevance and produces a personalized digest. Separately, modify a watched Confluence page and verify the assistant detects the change, classifies it as substantive, summarizes it, and assesses impact on the user.

**Acceptance Scenarios**:

1. **Given** 20 org-wide communications arrive in a week, **When** the weekly digest is generated, **Then** each item is classified as must-read, should-read, nice-to-know, or skip — personalized to the user's role and projects.
2. **Given** a watched architecture document adds a new "Service Mesh Migration" section, **When** the change is detected, **Then** the assistant produces a summary including what changed, what was added/modified/removed, and the impact on the user.
3. **Given** a security policy document changes to require secrets rotation every 90 days but a related runbook still describes a manual process, **When** the consistency checker runs, **Then** the assistant flags the contradiction between the two documents.
4. **Given** a change is classified as cosmetic (typo fix), **When** the change is detected, **Then** the user is not notified (cosmetic changes are suppressed per policy).

### User Story 6 — Visualization and Mind Mapping (Priority: P6)

A knowledge worker asks the assistant to visualize their task dependencies as an interactive graph, or to build a mind map of everything related to a specific project. The assistant gathers data from Jira, GitHub, Slack, email, and documents, organizes it thematically, and renders an interactive visualization.

**Why this priority**: Visualization enhances decision-making but is not critical to the core assistant workflow. It depends on the task intelligence and cross-system data from earlier phases.

**Independent Test**: Ask the assistant to generate a task dependency graph for a user with 10+ active tasks across Jira and GitHub. Verify it renders as an interactive visual with correct dependency relationships. Ask for a project mind map and verify it organizes related information from multiple sources by theme.

**Acceptance Scenarios**:

1. **Given** a user has tasks with known dependencies (PR blocks ticket, ticket blocks release), **When** the user requests a task dependency graph, **Then** an interactive visual is rendered showing tasks as nodes with dependency edges, and the user can click nodes for details.
2. **Given** a user requests a mind map for "Q2 Data Platform Migration," **When** the assistant gathers context, **Then** it organizes related Jira tickets, PRs, Slack threads, design docs, and email threads by theme (not by source system).
3. **Given** a user asks "What should I focus on?", **When** the assistant generates a priority matrix, **Then** it renders an Eisenhower matrix with tasks plotted by urgency and importance based on real signals.

### User Story 7 — Enterprise Administration and Policy Management (Priority: P7)

An enterprise admin configures the policy hierarchy (enterprise / org / team / user), defines model routing policies, action autonomy levels, integration permissions, and audit retention. Changes are validated, versioned, hot-reloaded within 60 seconds, and logged with who-changed-what-when-why. Admins can query the audit log to answer "what happened?" for any user and any time period.

**Why this priority**: Admin capabilities are essential for production deployment but are not needed for the core user-facing features to function. The policy engine itself is built in P1; this story adds the management layer on top.

**Independent Test**: As an enterprise admin, create a model policy that blocks external model calls for confidential data. Verify the policy takes effect within 60 seconds. As an org admin, attempt to create a policy that expands beyond the enterprise ceiling. Verify it is rejected. Query the audit log for a specific user's actions over the past week. Verify the results are complete and include every action, data access, model call, and policy decision.

**Acceptance Scenarios**:

1. **Given** an enterprise admin creates a policy blocking OpenAI models, **When** a user's assistant attempts to route a request to GPT-4, **Then** the request is denied and routed to an allowed model instead.
2. **Given** an org admin creates a policy allowing auto-send for emails, but the enterprise policy blocks auto-send, **Then** the enterprise policy takes precedence and auto-send remains blocked.
3. **Given** a policy change is saved, **When** 60 seconds have elapsed, **Then** the updated policy is active and governing all assistant actions.
4. **Given** an admin queries the audit log for "all actions by user X in the past 24 hours," **Then** every action is listed with timestamp, action type, data accessed, data classification, model used, policy applied, and outcome.

### Edge Cases

- What happens when a connector API is temporarily unavailable during briefing generation? The assistant MUST produce a partial briefing with a clear notice of which data sources were unreachable.
- What happens when two tasks from different systems appear to be the same but are ambiguous? The assistant MUST err on the side of showing both with a "possibly related" indicator rather than silently merging.
- What happens when a user revokes OAuth consent for a connector mid-session? The assistant MUST immediately stop accessing that connector, update the briefing to reflect the missing data source, and notify the user.
- What happens when the policy engine restarts while an action is in-flight? The action MUST be denied (fail closed) and retried after the policy engine is available.
- What happens when an agent-to-agent exchange receives a malformed or missing OCIP header? The receiving agent MUST treat the message as human-generated and not auto-respond as an agent.
- What happens when a document change is detected but the previous version is not in cache? The assistant MUST fetch the full document and flag that no diff is available — only the current state.

## Requirements *(mandatory)*

### Functional Requirements

**Policy Engine**

- **FR-001**: System MUST evaluate every tool invocation against the policy engine before execution.
- **FR-002**: System MUST enforce a policy hierarchy where lower levels can restrict but never expand beyond the parent level (enterprise / org / team / user).
- **FR-003**: System MUST support four graduated autonomy levels for every action: autonomous, notify, approve, block.
- **FR-004**: System MUST deny all actions when the policy engine is unreachable (fail closed).
- **FR-005**: Policies MUST be declarative documents (not code) that are human-readable, version-controllable, and diffable.
- **FR-006**: Policy changes MUST take effect within 60 seconds via hot-reload.
- **FR-007**: Every policy change MUST be logged with who, what, when, and why.

**Data Classification**

- **FR-008**: System MUST classify all data it accesses with a classification level (public, internal, confidential, restricted). Classification MUST be assigned using per-connector defaults (e.g., Gmail defaults to "internal," public GitHub repos default to "public"), with AI-based reclassification to detect sensitive content that exceeds the default, and admin override capability.
- **FR-009**: Data classification MUST propagate through all processing — a summary of an internal document produces an internal summary.
- **FR-010**: Data classified above "internal" MUST NOT be sent to external model providers unless explicitly allowed by enterprise policy.
- **FR-011**: Raw user data (email bodies, message content) MUST be processed and discarded — only structured extractions (tasks, summaries, classifications) are persisted.

**Connectors and Task Intelligence**

- **FR-012**: System MUST support connectors for email (Gmail, Outlook), calendar (GCal, Outlook), issue tracking (Jira, Linear, GitHub Issues), documents (GDrive, Confluence, Notion), and code (GitHub, GitLab). MVP release MUST include Gmail, GCal, Jira, GitHub, and GDrive. Remaining connectors (Outlook, Linear, Notion, Confluence, GitLab) are deferred to post-MVP releases.
- **FR-013**: Every connector MUST be read-only by default; write access MUST require explicit policy authorization.
- **FR-014**: System MUST discover tasks across all connected systems and correlate duplicates (same work item referenced in Slack, email, and Jira appears as one task).
- **FR-015**: System MUST score task priority using urgency signals: deadlines, sender seniority, follow-up frequency, SLA timers, and blocking relationships.
- **FR-016**: System MUST generate a daily briefing containing a prioritized task list, time-block suggestions, auto-response summary, and alerts.

**Auto-Response**

- **FR-017**: System MUST classify incoming messages as critical, needs-response, informational, or noise.
- **FR-018**: Auto-responses MUST clearly disclose AI involvement (configurable label such as "User's OpenClaw" or explicit AI disclosure).
- **FR-019**: Users MUST be able to review all auto-responses sent on their behalf, including the original message, generated response, and authorizing policy.
- **FR-020**: Auto-response scope MUST be configurable per channel, per contact, and per message classification.

**Work Tracking**

- **FR-021**: System MUST detect code activity events (PR opened, merged, commits pushed) via webhooks and correlate them with linked issue tracker tickets.
- **FR-022**: System MUST add summary comments to linked tickets and transition ticket status when policy allows.
- **FR-023**: System MUST generate end-of-day standup summaries from actual activity across all connected systems.

**Agent-to-Agent Protocol (OCIP)**

- **FR-024**: Every assistant-generated message MUST carry machine-readable OCIP metadata including message type, source agent identity, data classification, exchange round, and max rounds.
- **FR-025**: Classification MUST be enforced at the sender — data above the receiver's clearance MUST be filtered before transmission.
- **FR-026**: Every agent-to-agent exchange MUST have a maximum round count defined by policy; exceeding the limit MUST escalate to humans.
- **FR-027**: An agent MUST NOT commit its user to anything (meetings, deadlines, agreements) without human approval — this is structural, not configurable.
- **FR-028**: Both sides of every agent-to-agent exchange MUST be logged and reviewable by either user and by admins within their scope.
- **FR-028a**: Agent-to-agent exchanges MUST be allowed across organization boundaries within the same enterprise, governed by org-level policies that define what data can be shared and with whom. Cross-enterprise agent-to-agent exchanges MUST be blocked.

**Org Intelligence and Document Monitoring**

- **FR-029**: System MUST monitor configured org-wide communication channels and score each item for relevance to the specific user.
- **FR-030**: System MUST produce personalized digests (daily or weekly) with items classified as must-read, should-read, nice-to-know, or skip.
- **FR-031**: System MUST detect changes in watched documents, classify changes as cosmetic, minor, substantive, or critical, and summarize substantive changes.
- **FR-032**: System MUST assess the impact of document changes on each watching user based on their role, projects, and responsibilities.
- **FR-033**: System MUST detect inconsistencies between related documents when one of them changes.

**Visualization**

- **FR-034**: System MUST generate interactive task dependency graphs, priority matrices, and workload views.
- **FR-035**: System MUST generate mind maps from cross-system data organized by theme, not by source system.

**Security and Audit**

- **FR-036**: Enterprise deployment MUST require SSO/OIDC authentication — no password-only auth.
- **FR-037**: System MUST support four built-in roles: Enterprise Admin, Org Admin, Team Lead, User.
- **FR-038**: Every action, data access, model call, policy decision, and agent-to-agent exchange MUST be logged to an immutable, append-only audit log.
- **FR-039**: All data MUST be encrypted at rest and in transit.
- **FR-040**: User data MUST NOT be used for model training or fine-tuning.
- **FR-041**: Users MUST be able to export and delete all their data.

**Deployment**

- **FR-042**: System MUST be deployable on Kubernetes via a custom operator with CRD-based configuration.
- **FR-043**: System MUST support both shared multi-tenant and per-user instance deployment modes.
- **FR-044**: A new enterprise deployment for a 10-person team MUST be completable in less than 4 hours including K8s, SSO, and basic policies.
- **FR-045**: System MUST support up to 500 concurrent users per deployment for the initial release.

### Key Entities

- **Policy**: A declarative document governing assistant behavior. Has a scope (enterprise/org/team/user), a domain (models/actions/integrations/agent-to-agent/features/data/audit), and version history. Policies cascade hierarchically — lower levels restrict, never expand.
- **Task**: A unit of work discovered from one or more connected systems. Has a priority score, source references (which systems it was found in), correlation ID (linking duplicates), status, and deadline. Tasks are created by discovery, not manual entry. Lifecycle: discovered → active → completed → archived → purged. Completed tasks are archived after 30 days and purged after 90 days. Active task retention is 90 days.
- **Connector**: A connection to an external system (Gmail, Jira, GitHub, etc.) with scoped permissions (read/write/admin). Connectors are governed by integration policies and produce audit events for every access.
- **Agent Identity**: A machine-readable identity card for an OpenClaw Enterprise assistant instance. Includes user, org unit, capabilities (can receive queries, can auto-respond, can make commitments), and policy constraints. Used in OCIP exchanges.
- **Exchange**: An agent-to-agent conversation with OCIP metadata. Has participants, exchange type, round count, classification level, outcome (resolved/escalated), and full transcript. Logged on both sides.
- **Audit Entry**: An immutable record of an assistant action. Includes timestamp, user, action type, data accessed (with classification), model used (with token count), policy applied, and outcome. Append-only, never updated or deleted.
- **Briefing**: A generated daily summary for a user. Contains prioritized tasks, time-block suggestions, auto-response summaries, org news items, and document change alerts. Refreshed throughout the day as context changes.
- **Data Classification**: A label (public/internal/confidential/restricted) attached to every piece of data the assistant accesses. The classification propagates through all processing and determines model routing and sharing permissions.

## Assumptions

- OpenClaw's existing plugin system (registerTool, registerHook, registerService, registerHttpRoute, registerGatewayMethod, registerContextEngine) is sufficiently extensible for all enterprise features without requiring upstream core modifications.
- OpenClaw's existing `before_model_resolve` and `before_prompt_build` lifecycle hooks provide adequate interception points for policy enforcement and data classification injection.
- OpenClaw's existing `sessions_send` inter-session communication can be extended with OCIP metadata without requiring upstream protocol changes (or the upstream team will accept the enhancement proposal).
- Multi-tenancy will initially use a multi-gateway approach (one OpenClaw Gateway instance per tenant managed by a K8s operator) rather than deep session isolation within a single gateway.
- OPA (Open Policy Agent) or Cedar will be used for policy evaluation, called from the plugin layer via REST/gRPC. OPA runs as a sidecar alongside each gateway instance.
- Connector OAuth tokens are stored in a secrets manager (K8s Secrets or HashiCorp Vault), never in configuration files.
- The system processes raw user data (email bodies, message content) ephemerally and persists only structured extractions.
- Lobster (OpenClaw's workflow engine) handles complex multi-step enterprise workflows rather than building a custom workflow engine.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: An enterprise can complete a full deployment (K8s, SSO, basic policies, 3+ connectors) in under 4 hours from scratch.
- **SC-002**: Users report net time savings within 2 weeks of adoption, measurable through the assistant's own metrics (tasks auto-completed, auto-responses sent, briefing usage).
- **SC-003**: The daily briefing accurately correlates and deduplicates tasks across 5+ connected systems with less than 5% false-positive deduplication rate.
- **SC-004**: Auto-response classification accuracy exceeds 90% (measured by user correction rate — users override or retract fewer than 10% of auto-responses).
- **SC-005**: Zero data leakage incidents caused by classification enforcement failures — no data above "internal" reaches an unauthorized model or agent.
- **SC-006**: Policy changes take effect within 60 seconds of being saved.
- **SC-007**: The audit log can answer any "what happened?" question about any assistant action, for any user, within the retention period, in under 10 seconds query time.
- **SC-008**: Agent-to-agent exchanges never produce infinite loops — all exchanges terminate at or before the configured round limit.
- **SC-009**: Every enterprise feature works with OpenClaw's existing messaging channels without requiring channel modifications.
- **SC-010**: Upstream OpenClaw upgrades do not break the enterprise layer, validated by CI tests against each upstream release.
- **SC-011**: Every plugin achieves greater than 80% unit test coverage.
- **SC-012**: Users can review 100% of actions taken on their behalf, including auto-responses, Jira updates, and agent-to-agent exchanges, through the activity log.
